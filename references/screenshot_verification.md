# Screenshot Verification Protocol

> 修正の正確さを3層で検証し、設計観点とユーザー目線の2軸で品質を判定する。
> HARD-GATE: HG-2 — 全3層パスするまで ReviewNote を「完了」にしない。

## 3-Layer Verification

修正適用後、必ず以下の3層すべてで get_screenshot を実行する。

### L3: Component Level

```
get_screenshot(nodeId=修正ノードID, fileKey=ファイルキー)
```

**確認観点**:
- 修正が正確に反映されているか（色、テキスト、サイズ等）
- 文字化け・□プレースホルダーがないか（フォント読み込み失敗）
- テキスト切り詰めがないか（日本語幅の問題）
- 要素の重なりがないか（auto-layout 崩壊）

### L2: Frame Level

```
get_screenshot(nodeId=親フレームID, fileKey=ファイルキー)
```

**確認観点**:
- 修正ノードが周辺要素と調和しているか
- 余白バランスが崩れていないか
- 色の統一感が保たれているか
- 情報の階層構造が維持されているか

### L1: Page Level

```
get_screenshot(nodeId=ページID, fileKey=ファイルキー)  // ページ全体
```

**確認観点**:
- ページ全体のバランスが崩れていないか
- 他のフレームとの整合性が保たれているか
- orphan ノード（浮遊要素）が発生していないか

## 品質判定: 設計観点（5項目）

Design Personality を基準に判定する。

| # | チェック項目 | 基準 | 失敗例 |
|---|------------|------|--------|
| D1 | カラートークン準拠 | ui_design_brief.md の Colors Table と一致 | パレット外の #FF0000 が使われている |
| D2 | タイポグラフィ準拠 | brief の Typography Table と一致 | Body に H1 サイズが使われている |
| D3 | スペーシング準拠 | 4px/8px グリッドに乗っている | 余白が 13px（非グリッド値） |
| D4 | ビジュアルヒエラルキー | hero 要素が最も目立つ | 修正で補助要素が hero より目立ってしまった |
| D5 | アクセシビリティ | コントラスト比≥4.5:1, タッチ≥44px | 薄いグレーテキスト(#CCC)が白背景に |

## 品質判定: ユーザー目線の違和感（7項目）

実際のユーザーがこの画面を見たときに感じる違和感を検出する。
Design Personality の targetUser を念頭に判定する。

| # | チェック項目 | 問い | 失敗例 |
|---|------------|------|--------|
| U1 | 混乱リスク | このUI要素の意味が一目で分かるか？ | ボタンなのかラベルなのか判別できない |
| U2 | レイアウトの自然さ | ターゲットユーザーの期待する配置か？ | CTAが画面下端に埋もれている |
| U3 | 情報優先度 | 最も重要な情報が最も目立っているか？ | 価格より注釈の方が大きく表示 |
| U4 | アフォーダンス | クリック/タップ可能な要素がそう見えるか？ | テキストリンクが本文と同じスタイル |
| U5 | 一体感 | 修正箇所が「継ぎはぎ」に見えないか？ | 色だけ変わって周辺のトーンと不一致 |
| U6 | 日本語可読性 | テキストが読みやすいか？ | 行間が狭い、文字が小さすぎる、切り詰め |
| U7 | ブランド一貫性 | 同じプロダクトの画面に見えるか？ | 修正後だけ別アプリのような見た目 |

### U6 日本語可読性の具体基準

- 本文フォントサイズ: ≥14px
- 行間 (lineHeight): 170-180%
- 段落間余白: ≥12px
- テキスト切り詰め: textAutoResize = "HEIGHT"（横方向は親の FILL）
- 禁則処理: 句読点が行頭に来ていないか（Figma 上での目視）

## 判定フロー

```
L3 screenshot → D1-D5 + U1-U7 チェック
  ├── 全項目 PASS → L2 へ進む
  └── いずれか FAIL → 自動修復 (最大2回) → 再チェック
        └── 2回失敗 → ユーザーに報告、L2/L1 スキップ

L2 screenshot → D4 + U1-U5, U7 チェック（L2 固有の周辺調和）
  ├── 全項目 PASS → L1 へ進む
  └── いずれか FAIL → 自動修復 (最大2回) → 再チェック

L1 screenshot → 全体バランス目視確認
  ├── PASS → ReviewNote ステータスを「完了」に更新
  └── FAIL → ユーザーに報告
```

## 自動修復パターン

| 検出問題 | 修復アクション |
|---------|--------------|
| □プレースホルダー文字 | G-002 フォントフォールバックで再設定 |
| テキスト切り詰め | textAutoResize = "HEIGHT" + 親を HUG に |
| 要素の重なり | auto-layout の itemSpacing を確認・修正 |
| コントラスト不足 | テキスト色を darker に調整（パレット内で） |
| グリッド非準拠の余白 | 最寄りの 4px グリッド値に丸める |

自動修復後は必ず再度 get_screenshot を実行し、修復が成功したか確認する。

## 出力フォーマット

各 ReviewNote 処理後、ユーザーに以下を報告:

```
📋 ReviewNote #1: "SC-001のHeaderを赤に"
  対象: WF-US001 > Layout > Header
  修正: M-001 Color → fills #EF4444 (Error color from palette)
  検証:
    L3 Component: PASS (D1-D5 ✓, U1-U7 ✓)
    L2 Frame:     PASS (周辺との調和 ✓)
    L1 Page:      PASS (全体バランス ✓)
  ステータス: 未読 → 完了 ✅
```
