# figma-refine

> [Japanese version](README.md)

A Claude Code skill for context-aware Figma design refinement with AIDesigner integration. Understands your project's background (brand, target users, competitors, domain terminology), classifies modifications into Level A (property changes) and Level B (structural improvements), and refines Figma designs with optional AI-powered enhancements.

## What's New in v2.0

- **Level A/B Classification**: Automatically classifies modifications into property changes (Level A) and structural improvements (Level B)
- **AIDesigner MCP Integration**: Level B modifications leverage AIDesigner's `refine_design` to generate improved design proposals
- **Deep Scan**: Analyzes entire Figma file structure and captures Before screenshots before modifications
- **Design Change Report**: Analyzes modifications against designs/ definitions and recommends updates
- **Before/After Comparison**: Visual comparison for Level B modifications
- **No designs/ Fallback**: Asks users questions to manually build Design Personality

## Overview

- Loads project context from `designs/` folder (builds a "Design Personality")
- Detects ReviewNote components on Figma canvas and classifies into Level A/B
- Level A: Applies property modifications via Figma MCP with project-specific awareness
- Level B: Generates improved designs with AIDesigner → applies after user approval
- Outputs Design Change Report analyzing divergence from designs/
- 3-layer screenshot verification (Component → Frame → Page) × dual-axis quality judgment

## Workflow

```
Step 1: Context Load   — Read designs/ and build Design Personality
                         No designs/ → ask user questions for manual construction
Step 2: Deep Scan      — Analyze Figma file structure, capture Before screenshots
Step 3: Classify       — Classify ReviewNotes into Level A/B → apply Level A
Step 4: AIDesigner     — Level B improvements via AIDesigner → user approval
Step 5: Report         — Design Change Report (divergence analysis + update recommendations)
Step 6: Verify         — 3-layer × dual-axis screenshot verification + Before/After
```

## Level A/B Classification

| Level | Scope | Processing |
|-------|-------|-----------|
| **A** (Property) | Color, text, spacing, etc. | Direct modification via M-001 to M-007 |
| **B** (Structural) | Layout changes, abstract instructions ("make it modern") | AIDesigner refine_design + user approval |

## How to Place ReviewNotes

Place a ReviewNote instance near the element you want to modify, then write the modification instructions in the Content field.
Filling in the TargetHint field with the target frame or node name improves Name Mode resolution accuracy.

## Permission Model

| Operation | Permission |
|-----------|-----------|
| designs/ read | Free |
| designs/ write | **Prohibited** |
| designs/ update proposal | Output via Design Change Report |
| designs/ actual update | Delegate to `/requirements_designer enhance` or manual |

## Supported Modifications

| Code | Category | Examples |
|------|----------|---------|
| M-001 | Color | Fill/stroke color change |
| M-002 | Text | Content, font size, font weight |
| M-003 | Size | Width/height (non-auto-layout only) |
| M-004 | Corner | Corner radius |
| M-005 | Spacing | Padding, gap (auto-layout frames) |
| M-006 | Visibility | Show/hide |
| M-007 | Opacity | 0.0 to 1.0 |
| Level B | Structural | Layout changes, abstract design improvements |

## Limitations

- Node creation/deletion not supported (structural changes require manual intervention)
- Image replacement requires image hash (not available via ReviewNote)
- Variant switching requires component set understanding
- Effects (shadow, blur) have complex parameters requiring visual confirmation
- Writing to designs/ is prohibited (divergence reported via Design Change Report only)
